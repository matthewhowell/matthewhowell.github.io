Slightly Nicer Footnotes for the Web

January 31, 2021

I read, primarily, on the Web. In that reading, I routinely encounter pieces of writing that include annotations.

And annotations are good.

They are a useful tool for writers, offering an efficient way support and clarify their words.

In traditional print media, annotations are often cumbersome. Footnotes require physical space at the bottom of pages and endnotes require additional pages. Both require a context switch from the reader that can interrupt a point or a narrative.

On the Web, though, annotations could be better! They should be better. The Web is a beautiful place for words and we should always be pushing toward a better reading experience.

Today, to that end, I am introducing Reasonable Footnotes.

Reasonable Footnotes is a small piece of free software that transforms most website footnotes into slightly nicer website footnotes.

For specifics, including a getting started guide, examples, and documentation, please visit the Reasonable Footnotes website.

And if you’re curious about the motivation for the project, please read on as I make an effort to answer the first question that I ask of myself whenever I try to make something.

Why does this exist?

Reasonable Footnotes exists to make website footnotes slightly nicer.

Reasonable Footnotes should make Web footnotes work better for more people.

Website footnotes, today

HTML does not include a dedicated footnote or annotation element. So, footnotes on the Web are website features, not browser features. They are not standardized and there are many ways of implementing HTML footnotes. One specific method is popular enough that I’m comfortable calling it the nearly default Web footnote implementation.

It looks something like this: a small superscript link (a reference mark) will jump the reader farther down the page to a Footnotes section. The reader can then read the footnote and then click another link to jump them back to the reference mark in the main text.

Why is Reasonable Footnotes better?

  1. It improves the user experience by displaying the footnote without forcing a context switch on the reader
  2. It improves accessibility by defaulting a handful of sane decisions
  3. It does no harm by progressively enhancing existing footnotes

Why might it be worse?

  • It requires JavaScript[^]
  • It does not work in older browser (specifically Internet Explorer)[^]
  • Reading experience, like most experience, is subjective. You may not like the way it works and looks.[^]

Features

Out of the box, with very limited fuss, Reasonable Footnotes should:

  • Work with existing footnote markup
  • Allow keyboard navigation of footnotes (traversing, opening, and closing)
  • Allow multiple footnotes to be open at once
  • Allow footnote content to be highlighted and copied
  • Include accessibility metadata in markup (titles, aria attributes)
  • Include reasonably attractive footnote styles
  • Include either inline or bottom
  • Include print styles for footnotes

For More People

The first problem with having lots of competing implementations of footnotes is that not all implementors are going to have the resources and motivation to take accessibility seriously.

For those making the argument that an HTML <footnote> element should exist, inaccessible footnote implementations should convince most reasonable people.

Reasonable Footnotes should never make existing less accessible and will forever try to make existing footnotes more accessible.

So, out of the box, without any fuss, Reasonable Footnotes:

  • Works with most existing footnote markup
  • Allows keyboard navigation (tab through footnote buttons, space bar to open)
  • Allows multiple footnotes to be open at once
  • Allows footnote content to be highlighted/copied
  • Includes print styles for footnotes

User Experience

Even responsible footnote markup can, and often will, present poor user experiences.

The most common footnote implementation on the Web is to use a pair of <a> elements. The reference mark (usually a superscript footnote number) links to a location farther down the page, scrolling the user to the footnote content. There, at the end of the footnote content the user will find a link that will return them to the reference mark.

This idea, which is in fact noted in the HTML spec itself, is workable, but unpleasant.

It feels more like an endnote in a long book, which require the reader to leave the page they’re on, locate a note in a different part of the book and then return.

It’s jarring. It also removes the note from its context.

Reasonable Footnotes should never make the reading experience worse and will forever try to make it nicer.
